## ⚙️ 算法核心设计原理

小车在台下接收到串口指令 `Y` 后，会进入**自主登台测试状态机**。小车通过旋转扫描周围环境，并根据测量数据自适应执行“边缘登台”或“角落避障脱困 + 登台”。

### 1. 270° 自转环境扫描与判定
小车在初始位置清零 MPU6050 陀螺仪偏航角（Yaw = 0°），随后逆时针（CCW）旋转 3 次 90°，在 4 个停顿点（Yaw 分别为 0°、90°、180°、-90°）记录**车尾高位红外传感器 (`IR_Distance_UP` / `ir_distance[7]`)** 的距离数值，存入数组 `up_ir_values[4]` 中。
*   **阻挡判定阈值**：`up_ir_values[i] < 95.0f` 时判定为该方向有墙壁阻挡。
*   **边缘判定 (Edge)**：若仅有 1 个方向被阻挡，判定为处于擂台边缘。
*   **角落判定 (Corner)**：若有两个环形相邻的方向被阻挡（即首尾或相邻索引同时被阻挡），判定为处于死角。

### 2. 闭环旋转控制与锁角
所有旋转动作采用**航向角（Yaw）闭环比例-微分 (PD) 控制**，控制律如下：
$$yaw\_err = IMU\_Data.Yaw - target\_angle\_lock$$
结合微分项实现平稳转向，防止电机满载过冲或抖动。
> [!NOTE]
> 偏航角负反馈极性为 `IMU_Data.Yaw - target`。

## 🔄 登台动作流程与数学推导

### 🟢 场景 A：处于边缘 (Edge)
小车只需正对围栏物理对齐，然后倒车登台。
1.  **确定方向**：被阻挡的方向为 `D_blocked`。由于探测时是车尾朝墙，所以正对围栏的车头方向为：
    $$climb\_target\_angle = (D\_blocked \times 90.0^\circ) + 180.0^\circ$$
2.  **物理对齐**：转向目标角，随后向前直行顶住围栏（`ROBOT_CLIMB_FORWARD_UNTIL_WALL`），利用平滑 Dios 样前铲与围栏物理撞击实现车身完美垂直对齐。
3.  **倒退冲台**：触发对齐条件（前红外距离变小或超时）后，延迟 1 秒使车身压实，随后全速向后倒车（`Motor_Control(0, -100)`）1 秒冲上擂台。

### 🟡 场景 B：处于角落 (Corner)
角落中包含两个垂直墙面：**通道末端墙壁**（通常贴得更近，计为 `D_smaller`）和**擂台围栏**（计为 `D_larger`）。小车必须先移出角落，再执行对齐。
1.  **第一阶段：角落脱困 (Stage 1)**
    *   **目标**：小车需要远离紧贴的通道末端墙壁（`D_smaller`）。
    *   **方向计算**：车头正对开阔方向的偏航角为：
        $$escape\_angle = D\_smaller \times 90.0^\circ$$
    *   **动作**：小车转身至该角度，向前直行行驶 1.5 秒（固定定时器控制），安全走出窄道角落，来到擂台边缘。
2.  **第二阶段：物理对齐与冲台 (Stage 2)**
    *   **目标**：正对围栏墙壁（`D_larger`）物理对齐并登台。
    *   **方向计算**：车头正对围栏墙壁的偏航角为：
        $$align\_angle = (saved\_D\_larger \times 90.0^\circ) + 180.0^\circ$$
    *   **动作**：小车第二次转身正对围栏，向前直行推死围栏完成对齐，随后延迟 1 秒，全速向后倒车 1 秒冲上擂台。

## 🛠️ 关键状态判定条件

在物理对齐阶段（`ROBOT_CLIMB_FORWARD_UNTIL_WALL`），为了防止因雷达/红外传感器超距输出 `999.0f` 导致误判，代码中加入了严格的**数据过滤与超时保护双重锁**：
```c
// 过滤 999.0f 等无效超距值，同时结合前红外近距离判断与 2 秒硬超时保护
if ((IR_Distance_F > 48.0f && IR_Distance_F < 150.0f) || 
    (IR_Distance_B > 60.0f && IR_Distance_B < 150.0f) || 
    (IR_Distance_F < 20.0f && IR_Distance_F > 1.0f) || 
    elapsed_align >= 2000U)
{
    // 触发对齐，准备倒车登台
}
```

## ⚠️ 硬件排障指南（针对电机/编码器故障）

在实车调试中，如果小车出现以下异常物理行为，通常是**硬件故障**而非软件算法问题：

### 1. 典型故障表现与排查

| 异常物理行为 | 可能的硬件故障原因 |
| :--- | :--- |
| **小车前行时极其缓慢、往前蠕动** | 某一侧车轮未正常旋转（相当于拖着刹车皮前行，阻力极大）。 |
| **小车往前直行时，明显往左或往右严重弯曲弯偏** | 左右发力不对称。一侧的电机没有工作。 |
| **全速向后倒车时，无法直行，而是瞬间甩尾快速自转一圈** | 单侧双轮正常向后退，但另一侧轮子停转。极大的扭矩差直接造成车体甩尾原地转圈。 |

### 2. 基于串口测速数据 (RPM) 精准定位
小车运行测试时，会通过串口以 `1Hz` 的频率打印四个电机的**归一化测速值**：
`RPM: [左前], [右前], [左后], [右后]` (数值范围为 `-100.0f` 至 `100.0f`)。

*   **正常情况**（如全速后退时）：`RPM: -67.4, -76.9, -70.5, -72.0`（四个值都应该在 `-60` 以下）。
*   **异常情况**（若左后轮不转）：`RPM: -67.4, -76.9, -1.5, -72.0`（第三个数值接近 0）。
    > [!CAUTION]
    > 若发现某一个轮子的 RPM 值在电机运转指令下持续在 `0` 附近跳动，说明**该电机未工作**。请立即检查：
    > 1. 该电机的排线和电源插头是否松动或脱落。
    > 2. 该通道的电机驱动板（如 MOS管、控制引脚）是否损坏。
    > 3. 电机本身的减速箱或转轴是否被异物卡死。
