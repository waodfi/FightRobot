# STM32 格斗机器人：台下自主登台算法设计与实现方案 (二次优化测试版)

本项目在 [freertos.c](file:///d:/Project/STM32_Project/FightRobot/Core/Src/freertos.c) 中实现了一套完善的**台下自主登台算法**。该算法通过“物理对齐 + 陀螺仪锁角 + PID纠偏”三合一连招，确保小车能完美检测周围环境并直直地冲上擂台。

---

## 1. 软件架构设计

我们将在 `freertos.c` 中扩展现有的状态机，增加有关台下自主检测、扫描、脱困、顶墙和纠偏登台的相关状态。

### 扩展后的状态枚举 `RobotState_e`
```c
typedef enum {
    ROBOT_WAITING           = 0,   /* 等待中：等待用户用手遮挡左右红外 */
    ROBOT_ARMED             = 1,   /* 已就绪：等待用户松开双手 */
    ROBOT_LAUNCH_DECIDE     = 2,   /* 启动决策：自动读取底部灰度判定在台上还是台下 */
    ROBOT_CLIMBING_ONSTAGE  = 3,   /* 台上启动：直接静止不动 */
    ROBOT_SCANNING          = 4,   /* 台下扫描：原地缓慢自转，寻找边缘或角落 */
    ROBOT_CORNER_ESCAPE     = 5,   /* 角落脱困：根据相邻墙壁位置，执行前进/后退 + 90°旋转 */
    ROBOT_SQUARING          = 6,   /* 物理顶墙：向前行进顶住围栏，使车身绝对对齐，并重置IMU偏航角 */
    ROBOT_CLIMBING_OFFSTAGE = 7,   /* 闭环冲台：全速后退，并启用闭环角度比例纠偏(锁角0°), 直行登台 */
    ROBOT_FINISHED          = 8,   /* 登台/测试完成：保持绝对静止 */
    ROBOT_RUNNING           = 9    /* 运行状态：常规模式 */
} RobotState_e;
```

## 2. 深度物理调试与二次优化细节 (针对测试反馈)

### 2.1 解决上电前 1 秒由于传感器未稳导致自适应软启动失效
- **痛点**：上电初期 `Sensor_Task` 尚未启动完成，传感器可能读到初始 `0.0` 盲值，导致动态计算出的手遮挡/释放门限过小，从而使软启动在某些角度彻底失效。
- **优化**：在校准循环中加入了 **`800ms ~ 1s` 的延时缓冲**，确保传感器完成硬件上电稳定与首轮采样，并对计算出的动态门限进行了下限安全保护（最低 `12cm` / `15cm`），彻底根治软启动失效与死机问题。

### 2.2 解决“屁股快要碰到擂台小车就停了”的提早停机异常
- **痛点**：由于小车在倒退登台，后部灰度传感器最先越过擂台边缘。若使用“四个灰度均值”作为判定，在屁股刚越过边缘时，后灰度读数骤降，直接拉低均值提早触发了登台成功判定，导致前轮还没上去小车就提前停在坡上。
- **优化**：将登台成功判定标准升级为 **`Grey_Front < OFFSTAGE_GREY_THRESHOLD` (前灰度传感器检测到擂台面)**。
- **效果**：只有在整车完全爬过边缘、最前方的灰度传感器也越入擂台的一瞬间，才会精准触发登台成功，彻底根治爬坡到一半突然断电的现象！同时将超时时间设为更稳妥的 **`1.5秒 (1500ms)`**，确保有充足的时间在轮子打滑时完成攀爬。

### 2.3 解决在走廊死角/靠近台阶时一直打转不退避的判定异常
- **痛点**：在靠近台阶或侧面贴墙的角落自转时，原代码要求“非墙侧必须大于 200cm (均开阔)”，在小场地下这一条件太严苛，导致匹配失败，一直在那原地打转。
- **优化**：
  1. 将自转边缘扫描改为 **`左右红外至少一侧开阔 (|| 逻辑)`**，只要一侧开阔即允许判定，杜绝在走廊端头发生死转。
  2. 角落判定门限 `CORNER_OPEN_THRESHOLD` 优化为更适应小型场地的 **`80.0f cm`**，且将非墙侧的两个条件由 `&&` 升级为 **`只要有一侧开阔就允许脱困 (|| 逻辑)`**。这保证了无论在什么复杂的夹角中，小车都能立刻脱困并转向对齐。
